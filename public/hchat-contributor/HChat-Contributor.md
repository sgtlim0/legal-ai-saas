## 01. 프로젝트 개요

AI Execution Transformation Project (AETP)

> "AI를 '사용하는' 조직에서 AI를 '실행하고 확산하는' 조직으로"

| 축 | 내용 |
|---|------|
| 구조적 전환 | 개인 중심 AI 활용에서 조직 차원 실행 구조로 전환 |
| 생산성 혁신 | 반복 업무 자동화 + 시간 절감 효과 창출 |

| 역량 강화 | 자기경영 기반 AI 전문가 육성 |

**핵심 인프라:** H Chat (핵심 플랫폼) · Confluence (문서/대시보드) · GitLab (코드/버전관리)

## 02. H Chat 핵심 가치

### Enterprise AI Assistant

| 영역 | 효과 |
|------|------|
| Enterprise 보안 | 사내 인프라 내 구축, 데이터 외부 유출 Zero, 부서별 권한 관리 |
| 현업 맞춤 최적화 | 전사 문서 DB 연계, 현대오토에버 워크플로우 학습, 부서별 특화 템플릿 |

### H Chat vs 일반 ChatGPT

| H Chat | 일반 ChatGPT |
|--------|-------------|
| ✓ 사내 인프라 (보안 완벽) | ✗ 외부 서버 (보안 리스크) |
| ✓ 사내 문서 DB 연동 | ✗ 사내 문서 미연동 |
| ✓ 조직 라이선스 (비용 효율) | ✗ 개인 구독 (비용 증가) |
| ✓ 부서별 맞춤 템플릿 | ✗ 범용 템플릿만 제공 |

## 07. 핵심 철학적 기반

### 3가지 이론의 융합

**룬샷 이론** — "혁신은 아이디어가 아니라 구조의 문제다"
- P-Type 혁신 보호 → GitLab Experimental
- S-Type 실험 허용 → Branch 자유 실험
- 초기 취약성 보호 → 실패 허용 문화
- 점진적 확산 → Dev→Staging→Prod

**자기경영** — "지식근로자는 스스로를 경영해야 한다"
- 나의 강점은? → 강점 영역 AI 특화
- 무엇에 기여? → Confluence 목표 설정
- 피드백 분석? → Before/After 기록
- 성과 기록? → GitLab Issue 추적
- 지속 개선? → MR 피드백

**R.U.N.SHOT** — 실행 문화
- **R**un — H Chat 즉시 실험
- **U**nderstand — Confluence 기록
- **N**urture — GitLab Branch 보호
- **S**cale — Production 배포
- S**HOT** — Public Repo 공개

> 룬샷의 구조 + 자기경영의 기록 + R.U.N.SHOT의 실행 = H Chat 컨트리뷰터

## 10. 운영 프레임워크

### 프롬프트 제작 → 검증 → 배포 플로우

1. **로컬 실험** — H Chat 테스트 3회 이상, 다양한 입력값 시도
2. **GitLab 커밋** — Fork & Branch, 마크다운 작성 & Push
3. **Merge Request** — MR 생성, Reviewer 지정
4. **CI/CD 검증** — 문법 체크, 토큰 계산, 보안 스캔, 일관성 테스트
5. **코드 리뷰** — 운영팀 품질 평가, 재현성 확인, 보안 점검
6. **Merge & 배포** — dev→staging→main, H Chat 자동 배포

### 품질 게이트 (Quality Gate)

| 게이트 | 기준 |
|--------|------|
| 토큰 세이버 | 4,000 토큰 이하 (비용 효율성) |
| 보안 스캐너 | API Key/민감정보 차단 (TruffleHog) |
| 재현성 테스트 | 동일 입력 5회 실행, 일관성 측정 |
| 품질 점수 | 명확성/구체성/완성도 0.7 이상 |

## 11. 자동화 연동 시나리오

### 새 프롬프트 등록

1. 컨트리뷰터 GitLab 커밋
2. CI 파이프라인 자동 실행 (✓ 문법 ✓ 토큰 ✓ 보안 ✓ 일관성)
3. staging 자동 Merge
4. Confluence 알림 발송
5. 1주일 검증 후 main 승격
6. H Chat 템플릿 마켓 배포
7. "검증 완료" 배지 표시

### 프롬프트 피드백

1. 사용자 H Chat에서 사용 → 만족도 평가 (1-5점)
2. 평가 데이터 Confluence 전송, 평균 만족도 자동 업데이트
3. 4.5점+ → "High Quality" 태그 / 3.0점- → Issue 자동 생성

### 실패 사례 공유

1. GitLab Issue 생성 (failure) → Confluence 자동 동기화
2. "실패의 선물" 페이지 추가
3. 공감 10개+ → "인기 실패" 배지
4. 월간 "이달의 실패" → 포상

> GitLab Commit → CI/CD 검증 → Confluence 알림 → H Chat 배포 → 성과 측정 → Dashboard 업데이트

| 참여율 저조 | 높음 | 높음 | Shadow AI 양성화 인센티브 + 임원 스폰서 + 성공 사례 공유 + 실패 포상 문화 |
| 프롬프트 품질 미달 | 중간 | 높음 | CI/CD 자동 검증 + 운영팀 코드 리뷰 게이트 + 재현성 테스트 5회 필수 |
| 보안 리스크 | 중간 | 높음 | TruffleHog 시크릿 스캔 + 민감정보 마스킹 + 사내 인프라만 사용 + 감사 로그 |
| LLM 모델 변경 | 중간 | 중간 | model_version 태깅 필수 + 프롬프트 GitOps 관리 + 버전별 호환성 테스트 |
| 조직 저항 | 중간 | 중간 | 점진적 도입 (10명→30명→전사) + R.U.N.SHOT 문화 연계 + 챔피언 육성 |

### 핵심 원칙
- 실패를 처벌하지 않고 **포상**합니다
- 모든 프롬프트는 CI/CD 검증 + 코드 리뷰를 통과해야 배포
- 감사 로그로 모든 활동 추적
- 컨트리뷰터는 자율적 참여, **강제가 아닌 동기부여**
